Parallel branch environments

Every git worktree gets its own running stack.

StackMux turns parallel branches into isolated, testable environments with deterministic ports, Docker orchestration, and per-worktree runtime config. Git gives you the branches. StackMux gives you live environments you can actually use at the same time.

Test without merging. See exactly what each branch does before integration.
Isolate every workstream. Give each developer or agent its own running environment.

What it looks like

$ stackmux start feat/auth
claim slot 1
create git worktree for feat/auth
render stack files and agent config
start services on isolated ports

Main: localhost:3000
feat/auth: localhost:3001
fix/bug: localhost:3002
Main Base stack localhost:3000
Slot 1 feat/auth localhost:3001
Slot 2 fix/bug localhost:3002

One command creates a new worktree, assigns a slot, and makes the branch live on its own local URL.

The problem is not git. The problem is everything around it.

Git worktrees already solve parallel checkouts. The missing layer is the runtime: ports, services, database separation, generated config, and a repeatable workflow for humans and coding agents. StackMux fills that gap.

What StackMux adds

The current design centers on predictable slots, generated environment files, and optional Docker service isolation that scales from small apps to multi-service repos.

Deterministic ports

Base app on :3000, slot 1 on :3001, slot 2 on :3002, and so on. The same pattern extends across every configured service.

Rendered stack files

Templates generate Compose overrides and per-worktree configuration with the right paths, ports, and branch metadata already filled in.

Monorepo stack selection

Named stacks let one workstream run the notes slice, another run tasks only, and a third run the full platform without conflicts.

Expected workflow

This reflects the current reviewed README: claim a slot, create a worktree, render stack config, and run branches side by side.

01

Claim a slot

Each workstream gets a numbered slot that determines its port offsets and runtime identity.

02

Create the worktree

StackMux builds on git worktrees so each branch has its own directory and can run independently.

03

Render config

Templates generate per-worktree files for ports, Compose overrides, and agent context.

04

Run side by side

Multiple branches stay live at once, each on its own ports, without disturbing the main checkout.

Quick start and demo

The current README now covers the full public flow, from trying the demo to installing StackMux and starting real workstreams in your own project.

Try the demo

Start with the interactive demo to see slot allocation, service selection, and init flow without touching an existing project. 

git clone https://github.com/stackmux/stackmux.git
cd stackmux
./demo.sh
./demo.sh 1

Real project flow

Install the CLI, initialize a project once, then create and run isolated workstreams as needed. 

stackmux init
stackmux start feat/auth
stackmux status
stackmux finish feat/auth
